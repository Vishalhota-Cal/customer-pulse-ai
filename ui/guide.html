The feedback pile is infinite.
The reading time is not.

This is the working file for Port·05: what the mission actually asked for, what real companies run into at global scale, where today's tools quietly fall short, and the advanced instruments that close each gap.

The Brief
  1. Product/CX teams get hundreds of reviews, tickets, and survey responses weekly — too much to read manually.
  2. Build a system that auto-categorizes every piece of feedback.
  3. Detect sentiment on each item.
  4. Surface recurring, specific themes across all of it — not vague buckets.
  5. Generate a weekly insight summary, test on real samples, document accuracy, and submit the repo.
01

Real-World Pain Points

What CX and product teams actually run into, at companies operating far beyond a 34-row sample file.

Volume

Feedback overload

Large consumer apps see thousands of reviews and tickets weekly, across dozens of channels. Nobody on the team has time to read it all — the backlog just grows.

Language

English-first tooling, non-English world

Most feedback-analytics builds — including early versions of this one — are tuned on English. Globally, most customer feedback isn't in English at all.

Nuance

Sarcasm and cultural tone

"Oh great, another update that broke everything" reads as positive to a naive model. Irony is culturally specific and remains genuinely unsolved.

Follow-through

The insight-to-action gap

Teams get a nice weekly chart, and nothing changes after. Dashboards rarely connect an insight to an owner, a status, or a deadline.

Privacy

PII hiding in plain text

Real feedback often contains emails, order numbers, even partial payment details. Sending raw text to a third-party AI API risks GDPR, LGPD, and CCPA exposure.

Cost

Scale breaks the naive approach

Three sequential AI calls per item is fine at dozens of items. At fifty thousand a week, that's 150,000 calls — and a very different bill.

Drift

Models go stale quietly

Slang, products, and customer language shift. Without periodic re-validation, "what counts as urgent" silently drifts and nobody notices.

Signal loss

Alert fatigue

Flag too cautiously and "urgent" gets attached to everything. Once every item looks urgent, the label stops meaning anything to the team.

02

What Existing Platforms Miss

What most commercial feedback tools actually do, next to what they conspicuously leave out.

What most tools doClassify and score each ticket individually, one at a time.
What's missingCross-item theme clustering and trend-over-time — the aggregate pattern, not just the single ticket.
What most tools doRender a static dashboard of counts and charts.
What's missingOwnership and workflow tracking — a way to mark "we're fixing this" and see it through.
What most tools doShip NLP tuned overwhelmingly for English text.
What's missingThe same rigor — few-shot examples, confidence calibration — applied to non-English feedback, not just a flag-and-skip.
What most tools doSurface a single AI-generated health or sentiment score.
What's missingA documented, inspectable formula — executives trusting a number they can't see the math behind is a real adoption blocker.
What most tools doRequire manual CSV export/import between systems.
What's missingNative connectors to where feedback actually lives — app store reviews, ticketing systems, survey tools — wired in out of the box.
What most tools doBill flatly per seat or per month, cost hidden from the user.
What's missingVisibility into what each AI call actually costs, so usage can be tiered and controlled as volume grows.
03

Advanced Tooling Map

Where each real-world gap gets closed, and specifically where it would slot into this codebase's layers.

ProblemAdvanced Tool / TechniqueWhere It Fits Here
Multilingual coverage
Translation API or multilingual embeddings

Google Cloud Translation, Amazon Translate, or multilingual embedding models (Cohere Embed, OpenAI text-embedding-3).

Replace the current "detect non-English and flag" step in orchestration/pipeline.py with a translate-then-classify path, so non-English feedback gets real analysis instead of a skip.

True semantic theme clustering
Embeddings + vector database

pgvector, Pinecone, Weaviate, or Chroma for real semantic similarity, beyond string matching.

Swap brain/theme_aggregator.py's difflib string-similarity clustering for embedding-based clustering once volume outgrows a few hundred items a week.

Sarcasm and tone nuance
Chain-of-thought prompting / tone-specific fine-tunes

A reasoning pass that explicitly checks "is this sarcastic before scoring," or a sentiment model fine-tuned on ironic text.

Extend brain/sentiment.py's few-shot set with deliberately ironic examples, or add a second-pass tone check before the final score.

PII exposure
PII detection & redaction

Microsoft Presidio, AWS Comprehend PII detection.

Insert a redaction pass in the domain layer's validation step, before text ever reaches an external AI client.

Cost at real scale
Tiered models & prompt caching

A cheap/small model for first-pass triage; an expensive model only for ambiguous, low-confidence cases. Anthropic prompt caching for repeated system prompts.

Add tiered model selection to the AIClient construction in brain/classifier.py, escalating only when confidence is low.

Model drift
MLOps observability

Arize, LangSmith, or Weights & Biases, tracking confidence-score distributions over time.

Log every classification's confidence to a metrics store; alert when the rolling average confidence drifts downward.

No live alerting
Webhook / notification integrations

Slack API, PagerDuty, or an email digest via SendGrid.

Fire a webhook from orchestration/pipeline.py whenever a new High-urgency item is saved, instead of requiring someone to check the dashboard.

Fragmented data sources
Native connectors / iPaaS

Direct API ingestion from ticketing systems, app store review APIs, survey tools — or Zapier/Make for a no-code version.

New ingestion adapters that all feed the existing /api/feedback endpoint — the API doesn't care whether data came from a CSV, a script, or another system's live feed.

04

Cheat Notes for Demo Day

Fast, honest answers to the questions most likely to come up.

Q: Is this ready for global, enterprise-scale feedback volume?
Honestly, not yet. It's built and proven correct at demo scale. Getting to enterprise scale needs three specific things: real multilingual handling (not just detect-and-flag), tiered/cheaper models to control cost, and PII redaction before anything reaches an external API.
Q: What's the single biggest gap versus a commercial tool?
Workflow ownership. This surfaces what's wrong very well; it doesn't yet track who's fixing it or whether a flagged theme actually got resolved over time — beyond the basic Open/In Progress/Resolved status now in place.
Q: Why does the system flag non-English input instead of just classifying it anyway?
The few-shot examples in every prompt are English-only. Classifying a language the model was never shown examples of would mean confidently guessing — flagging it for a human is the safer, more honest failure mode.
Q: How would you prove this works on real-world data, not just the sample set?
Run it against a real, messy, mostly non-English dataset (e.g. the Olist Brazilian e-commerce reviews) — it's a fair stress test that would honestly reveal both the system's real strength (refusing to guess at an untrained language) and its current real limitation (English-only classification).
